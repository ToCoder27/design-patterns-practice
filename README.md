# 嵌入式设计模式实践

> 用嵌入式开发者的方式学习设计模式：**从痛点出发，用 C++ 实现，结合硬件场景**

---

## 学习路径

本教程按**难度**和**使用频率**分为三个阶段，建议按顺序学习：

```
┌─────────────────────────────────────────────────────────────┐
│  第一阶段：基础模式（必须先掌握）                             │
│  单例 → 策略 → 工厂                                         │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第二阶段：常用模式（工作中经常用）                           │
│  适配器 → 观察者 → 状态机 → 命令                             │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第三阶段：高级模式（复杂场景）                               │
│  模板方法 → 装饰器 → 外观 → 建造者                           │
└─────────────────────────────────────────────────────────────┘
```

## 模式对比速查

### 策略模式 vs 状态模式

| 对比项 | 策略模式 | 状态模式 |
|--------|----------|----------|
| **目的** | 切换算法 | 状态转换 |
| **关系** | 客户端设置策略 | 状态自己决定转换 |
| **场景** | 传感器驱动切换 | 售货机/协议状态机 |

### 适配器模式 vs 外观模式

| 对比项 | 适配器模式 | 外观模式 |
|--------|----------|----------|
| **目的** | 接口转换 | 简化接口 |
| **对象** | 一个被适配者 | 多个子系统 |

### 观察者模式 vs 命令模式

| 对比项 | 观察者模式 | 命令模式 |
|--------|----------|----------|
| **目的** | 一对多通知 | 封装操作 |
| **特性** | 自动通知 | 支持撤销/宏命令 |

## 设计原则

这些模式体现了以下设计原则：

| 原则 | 说明 | 体现模式 |
|------|------|----------|
| **开闭原则** | 对扩展开放，对修改关闭 | 策略、工厂、观察者 |
| **依赖倒置** | 依赖抽象，不依赖具体 | 工厂、命令、观察者 |
| **单一职责** | 一个类只负责一件事 | 命令、状态、装饰器 |
| **组合优于继承** | 优先使用组合而非继承 | 装饰器、策略、状态 |

## 学习建议

### ✅ 应该这样学

1. **从痛点出发**：先看"传统写法 vs 模式写法"，理解为什么要用这个模式
2. **结合硬件场景**：每个模式都有嵌入式实际应用场景
3. **动手实践**：编译运行代码，修改示例加深理解
4. **按顺序学习**：先基础后高级，前面的模式是后面的基础

### ❌ 不要这样学

1. 死记硬背类图
2. 只看不写
3. 为了用模式而用模式
4. 一开始就追求"完美设计"
